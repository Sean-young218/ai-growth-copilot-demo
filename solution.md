# Solution Notes

## 1. 方案概述

- 输入一条 B2B 潜在客户线索
- 加载 `lead-scoring-followup` Skill
- 调用 `query_product` 查询本地业务资料
- 让 LLM 基于资料输出线索评分、风险点和跟进建议
- 前端展示结果和 Trace

## 2. 关键设计理由


- **Skill**：沉淀可复用的业务流程和输出要求。
- **Tool**：把产品资料、销售 SOP、禁止承诺事项从模型生成中拆出来，减少幻觉。
- **Runner**：负责串联 Reasoning Summary、Act、Observe、Answer，形成可审计流程。
- **Trace**：让评审看到系统不是直接问 LLM，而是经过了资料查询和步骤记录。

## 3. ReAct 如何体现


- `Reasoning Summary`：根据线索判断是否需要查询产品资料。
- `Act`：调用 `query_product`。
- `Observe`：返回匹配产品、SOP、禁止承诺事项和来源。
- `Answer`：LLM 基于线索和观察结果生成最终建议。

## 4. 传给模型的上下文

- Skill 正文
- 用户输入的线索文本
- `query_product` 返回的产品资料
- 销售 SOP
- forbidden claims
- 输出章节要求和风险约束

也可以说明：没有把完整内部推理链暴露给前端，只展示审计摘要。

## 5. 风险控制

- 禁止编造价格、交付周期、客户案例、产品能力。
- 产品能力只来自 `product_catalog.json`。
- 销售动作参考 `sales_sop.md`。
- 涉及价格、周期、准确率、ROI、案例、合规承诺时要求人工复核。
- LLM 未配置或失败时返回本地兜底结果，并标注需要人工复核。

## 6. AI 使用说明

- 使用 AI 辅助梳理项目结构、文档表达和部分代码实现。
- 核心设计取舍由自己确认。
- 没有使用 Dify、Coze、n8n、Flowise 等低代码 Agent 平台作为主体实现。
- 最终代码和文档经过本地运行与测试验证。

## 7. 后续迭代

- 接入真实 CRM 或知识库。
- 把关键词匹配升级为向量检索或混合检索。
- 增加输出 schema 校验。
- 保存执行历史，方便销售主管复盘。
- 增加人工确认流程。
- 增加批量线索处理。

## 8. 有效性指标

- 销售是否采纳线索优先级建议。
- 首次跟进时间是否缩短。
- 跟进话术是否减少遗漏关键信息。
- 人工复核命中率。
- 误判率，比如高分线索无效、低分线索实际成交。
- 销售满意度或节省时间。
